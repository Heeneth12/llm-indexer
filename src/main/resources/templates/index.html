Why paste 1,200 lines for a 40-line change?

Say a file has 1,200 lines but the change only touches one 40-line method. Pasting the whole file wastes ~97% of those tokens — and buries the part that matters. llm-index sends the agent just that method, its callers, and its call chain. Same understanding, a fraction of the context, and the edit lands on the exact lines.

How it works

Four steps, no configuration: point it at code, it parses every file, writes the index, then you search it.

01

Point it at code

Run the CLI against a local project, or paste a git URL into the browser version above.

02

Parse every file

JavaParser walks each .java file and extracts classes, fields, methods, and Spring annotations.

03

Build the index

Writes tree/skeleton/dependency markdown plus a SQLite graph of USES, CALLS, CONTAINS, and EXPOSES edges.

04

Query it

Search a class or keyword and get matches, what depends on it, and its outward call chain — ready to paste into an LLM.

Precision over volume

The whole repo gets parsed and indexed — nothing is skipped. But a query only ever returns the exact file:line references that matter, not the repo itself. That's what keeps the result small enough to actually paste into a prompt.

Without an index
  • A 1,200-line file goes into the prompt for a 40-line fix
  • ~97% of those tokens are noise the model must skip past
  • The agent guesses which part to edit — and sometimes touches code it shouldn't
  • Context fills up fast; falls apart past a few hundred files
With llm-index
  • A query returns only the matching method's file:line refs — usually under 10% of the file
  • The agent opens exactly those lines and edits exactly that part, nothing else
  • Callers and call chain come along, so understanding stays intact
  • Works the same with any agent or LLM — the index is just markdown + SQLite it can read

One index, every tool

There's no plugin and no lock-in. The index is plain markdown and a SQLite file — anything that can read a file can use it.

The knowledge graph

Every class and method becomes a node; every Spring @GetMapping-style annotation becomes an endpoint node. Relationships between them become typed edges — this is the actual model written to graph.db:

ClassOrderService → CONTAINS → MethodOrderService.createOrder
MethodOrderService.createOrder → CALLS → MethodPaymentClient.charge
ClassOrderService → USES → ClassPaymentClient
MethodOrderController.create → EXPOSES → EndpointPOST /orders

A query walks these edges outward with a recursive SQL query, so "what does this call, and what does that call" resolves in one pass instead of N follow-up questions:

WITH RECURSIVE chain(name, depth) AS (
  SELECT name, 0 FROM nodes WHERE name LIKE 'OrderService%'
  UNION
  SELECT e.dst, c.depth + 1 FROM edges e
  JOIN chain c ON e.src = c.name
  WHERE e.predicate IN ('CALLS','USES') AND c.depth < 3
)
SELECT DISTINCT name, depth FROM chain WHERE depth > 0 ORDER BY depth;

It's a plain .db file — graph.db opens in any SQLite client if you want to write your own queries against nodes(name, type, file_path, line_no) and edges(src, predicate, dst, file_path).

What you get

Every build writes these into .llm-index/ next to your project.

  • 01-tree.mdDirectory structure, filtered to source files
  • 02-skeleton.mdEvery class's fields and method signatures, with line numbers
  • 03-dependencies.mdWhich classes use which — a flat dependency list
  • graph.dbSQLite knowledge graph (nodes/edges) you can query repeatedly
  • context.mdWritten on each query — matches, impact, and call chain for one term
  • INSTRUCTIONS.mdTells the LLM how to read the rest, so it doesn't ingest the whole repo

Get started

>_

Run it locally

Have Maven and Java 21+? Build the jar and point it at any project on your machine. The index is written straight into that project's .llm-index/ folder. 

./mvnw -q -DskipTests clean package
java -jar target/indexer-*.jar build .
java -jar target/indexer-*.jar query MyClassName

First command builds it, second indexes the current directory, third searches the graph and writes context.md.

Or point us at a repo
