Overview

llm-index turns a Java codebase into an LLM-friendly index: a directory tree, class/method skeletons, a dependency list, and a queryable SQLite knowledge graph. The whole repo gets parsed once; after that, a query for a class or keyword returns only the matching file:line references — typically under 10% of a file's raw code — instead of dumping the whole repo into a prompt.

It ships as a single jar with three front doors onto the same engine: a CLI, a web app, and a Maven library you can embed in your own Spring Boot app.

Architecture — how it's built

One shared indexing engine, three ways to reach it. Everything under com.llm.indexer:

  • core/TreeBuilder, SkeletonBuilder, DepsBuilder, GraphStore, Hashes, and IndexService — the engine. JavaParser walks every .java file; IndexService orchestrates the rest and is the single entry point both the CLI and web app call.
  • query/QueryService — graph traversal (matches, used-by, outward call chain via a recursive SQL CTE) shared by the CLI's query command and the web app's Query tab.
  • viz/GraphVisualizer — renders graph.db as a self-contained interactive HTML page (vis-network via CDN), shared by indexer visualize and the web app's Graph tab.
  • cli/CliRoot, BuildCommand, QueryCommand, VisualizeCommand — a picocli command tree. Runs with no Spring context at all, for fast startup.
  • web/WebController, IndexJobService, GitCloner, StartupIndexRunner/Properties, WebMvcConfig — Spring MVC + Thymeleaf. Manages job lifecycle (clone, index, evict) and everything under /llm-indexer/**.

A single IndexerApplication.main() decides which front door to use: if the first argument is build, query, or visualize, it runs the picocli command tree directly and exits — no web server, no Spring Boot startup cost. Otherwise it boots the full Spring Boot web app.

Data flow for a build: JavaParser parses each file → TreeBuilder/SkeletonBuilder/DepsBuilder produce the three markdown views → DepsBuilder also writes typed nodes (Class/Method/Endpoint) and edges (CONTAINS/USES/CALLS/EXPOSES) into graph.db via GraphStore. A SHA-256 hash cache (state.json) means rebuilds only re-parse files that actually changed.

Installation

CLI

./mvnw -q -DskipTests clean package
java -jar target/llm-index-exec.jar build .

Or grab a prebuilt jar from the Releases page instead of building it.

Maven / JitPack

<repositories>
  <repository>
    <id>jitpack.io</id>
    <url>https://jitpack.io</url>
  </repository>
</repositories>

<dependency>
  <groupId>com.github.Heeneth12</groupId>
  <artifactId>llm-indexer</artifactId>
  <version>v0.7.0</version>
</dependency>

Resolves to the plain llm-index.jar (not the executable one) — normal classpath, works like any library. See Embedding as a library below for the Spring wiring it needs.

Docker

docker build -t llm-indexer .
docker run -p 8080:8080 llm-indexer

Configuration (application.properties)

Auto-index a repo on boot — no CLI command, no pasting a URL. Every property below is prefixed llm-index.startup.

PropertyTypeDefaultDescription
enabled boolean false Master switch. Nothing else in this table does anything unless this is true.
path String — A local directory already on disk (e.g. a mounted volume) to index on boot.
repo-url String — Alternative to path: a git URL, cloned on first boot and fast-forward pulled on later restarts.
work-dir String system temp dir Where to clone repo-url into. Only used with repo-url, not path.
full boolean false Force a full re-parse every run, ignoring the SHA-256 change cache.
reindex-on-restart boolean true If false and an index already exists from a previous run, reuse it instead of rebuilding.
graph-visualization-enabled boolean false Also write .llm-index/graph.html (same output as indexer visualize) after each run.

Example — auto-index a mounted repo, reindex every restart, and keep a graph.html on disk:

llm-index.startup.enabled=true
llm-index.startup.path=/data/my-repo
llm-index.startup.full=false
llm-index.startup.reindex-on-restart=true
llm-index.startup.graph-visualization-enabled=true

Once configured, the result is always at /llm-indexer/jobs/startup and linked from the landing page. That job is exempt from the normal 30-minute eviction, and its directory is never auto-deleted — safe to point path at a real working repo.

CLI reference

CommandOptionsDescription
build [path] -o, --out <dir> (default .llm-index)
--full		 path defaults to the current directory. Indexes the project: tree, skeletons, dependencies, and the graph. --full forces a complete re-parse, ignoring the incremental cache.
query <term> -d, --dir <dir> (default .llm-index)
--hops <n> (default 3)		 Searches the graph for a class or keyword. Prints and writes context.md with matching code, what uses it, and its outward call chain up to --hops deep.
visualize -d, --dir <dir> (default .llm-index)
--filter <term>
--hops <n>
--no-open		 Generates .llm-index/graph.html and opens it, unless --no-open is set. --filter scopes to matching nodes; add --hops to also pull in nodes within N hops via CALLS/USES.

All three commands run without booting a web server or Spring context — the CLI is dispatched directly in IndexerApplication.main() when the first argument matches one of them.

HTTP API reference

Base path: /llm-indexer — everything (pages, CSS, API) lives under this one prefix, deliberately, so it can't collide with a host app's own routes when embedded as a library.

RouteParamsDescriptionExample
GET/llm-indexer/ — Landing page: paste-a-URL form, plus a link to the configured startup job if one exists. curl localhost:8080/llm-indexer/
GET/llm-indexer/docs — This page. —
POST/llm-indexer/index repoUrl Clones and indexes a public git repo, then redirects to its job page. curl -d repoUrl=https://github.com/o/r.git localhost:8080/llm-indexer/index
GET/llm-indexer/jobs/{id} — Job status and results: Tree / Skeleton / Dependencies / Graph / Query tabs. curl localhost:8080/llm-indexer/jobs/startup
POST/llm-indexer/jobs/{id}/query term, hops (default 3) Runs a graph query and re-renders the job page with matches, impact, and call chain. curl -d term=OrderService -d hops=2 .../jobs/{id}/query
GET/llm-indexer/jobs/{id}/graph filter, hops (optional) The interactive graph as a standalone HTML page — what the Graph tab embeds via iframe. curl ".../jobs/{id}/graph?filter=Order&hops=2"
GET/llm-indexer/jobs/{id}/file path, line (optional) Source file viewer, jumping straight to and highlighting the given line. curl ".../jobs/{id}/file?path=src/.../X.java&line=42"

Embedding as a library

llm-index's Spring beans (StartupIndexRunner, IndexJobService, and the web UI's WebController) live in com.llm.indexer.web — a different package tree than your own @SpringBootApplication class, so Spring's default component scan (rooted at your app's own package) won't discover them on its own. Point it there explicitly:

@SpringBootApplication
@ComponentScan(basePackages = {"your.own.package", "com.llm.indexer.web"})
@ConfigurationPropertiesScan(basePackages = "com.llm.indexer.web")
public class YourApplication {
    public static void main(String[] args) {
        SpringApplication.run(YourApplication.class, args);
    }
}

With that in place, llm-index.startup.* properties work exactly as documented above, and the full UI mounts at /llm-indexer/** in your app. It's safe alongside your own routes and static assets since everything is namespaced under that one prefix — nothing to exclude, nothing to reconfigure.

Deployment

The image is a plain container that listens on $PORT (defaults to 8080), so it runs on anything that runs a container — Render, Fly.io, Railway, a VPS behind Docker/nginx, and so on. No database is required; job state is in-memory and ephemeral by design (except a startup-configured job, which is permanent and never evicted). Push the repo (or just the built image), point your host at the Dockerfile, and set PORT if the platform requires a specific one.
